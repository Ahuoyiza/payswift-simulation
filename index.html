Context

PaySwift is a fintech startup building a mobile payments app for the Nigerian market. The product team needs to ship a P2P money transfer feature within 6 weeks. Ada (engineering) and Morgan (product) are working in parallel.

Your instructor walks through what Ada builds at each phase. Your job is to contribute what Morgan — the PM — should be doing at every step.

Bottleneck Watch
Vague PRD = blocked engineers
If Morgan doesn't specify the transaction limits, fee structure, or compliance rules clearly, Ada will make assumptions — and those assumptions may need to be rebuilt later at significant cost.

Timeline misalignment
If stakeholders expect 4 weeks and Ada says 6, Morgan must negotiate — not pressure the engineer. Cutting scope is safer than cutting quality in a fintech product.

Phase 3

Building the Backend

When the user taps "Confirm Transfer," here's what happens behind the scenes.

Ada's Engineering Tasks
  • 1
    Auth Service: Verify JWT token → check PIN → verify session not expired
  • 2
    Validation Service: Check sender active, recipient active, amount ≥ ₦100 ≤ ₦1M, daily limit not exceeded
  • 3
    Transaction Service: Lock wallets → debit sender → credit recipient → create record → COMMIT
  • 4
    Notification Service: Push to sender, push to recipient, queue email receipts
  • 5
    Response: Return success + transactionId to frontend

Bottleneck Watch
Undefined fee structure
Ada needs to know: does the sender pay a fee? If so, how is it calculated? This is a business decision, not a technical one — and it must come from Morgan before backend coding begins.
Bottleneck Watch
No rollback plan defined
If a transaction fails mid-way (network drop, server error), what happens to the user's money? Morgan must define the recovery policy before Ada can implement safe rollback behaviour.
Phase 4

Database & Data Flow

Ada designs the memory of the product — every user, wallet, and transaction stored safely.

Ada's Engineering Tasks
  • 1
    Design PostgreSQL schema: USERS, WALLETS, TRANSACTIONS tables
  • 2
    Implement ACID-compliant transaction processing with row-level locking
  • 3
    Write transaction flow: START → LOCK → VALIDATE → DEBIT → CREDIT → COMMIT (or ROLLBACK)
  • 4
    Set up Redis cache for fast balance lookups
  • 5
    Configure automated backups and disaster recovery

Bottleneck Watch
Schema changes after launch = expensive
Adding a column to a production database with millions of rows is a high-risk, time-consuming migration. Morgan must get the data requirements right before the first deploy — not after.
Bottleneck Watch
Compliance overlooked until audit
The CBN and NDPR have strict requirements for financial data retention in Nigeria. If Morgan doesn't factor these in early, the product may need costly refactoring to pass a regulatory audit.
Phase 5

Security Implementation

Ada layers in protection — encryption, authentication, fraud detection. All 5 layers at once.

Ada's Engineering Tasks
  • 1
    HTTPS/TLS: SSL cert, force HTTPS, strong cipher suites. Without it — anyone on WiFi sees your PIN
  • 2
    Authentication: JWT token generation, PIN hashing with Bcrypt, session expiry
  • 3
    Encryption at rest: AES-256 for bank details, Bcrypt for PINs, encrypted personal data
  • 4
    Rate limiting: Login 5/hr, transfers 10/min, balance checks 60/min, per-IP 100/min
  • 5
    Fraud detection: Risk score engine — new device (+15), unusual time (+5), large amount (+20), multiple recipients (+30)

Bottleneck Watch
Security added as an afterthought
Security can't be bolted on at the end. Fraud thresholds and lockout policies must be agreed before Ada builds the auth layer — otherwise they become expensive retrofits.
Bottleneck Watch
No fraud response process
When a transaction is flagged, who reviews it? What's the SLA? Without a defined process from Morgan, flagged transactions just pile up — and legitimate transfers get stuck.
Phase 6

API Integration

Ada ties it all together — the contract between frontend and backend, live and working.

Ada's Engineering Tasks

  • 2
    Implement POST /api/v1/transactions/transfer — full transfer flow
  • 3
    Implement GET /api/v1/users/search — recipient search with autocomplete
  • 4
    Implement GET /api/v1/wallets/balance — real-time balance
  • 5
    Write API documentation in Swagger/OpenAPI format
  • 6
    Test all endpoints end-to-end in staging environment

Skipping UAT before launch
Morgan must run User Acceptance Testing before any public launch. Releasing untested code in a payments product can result in real users losing real money — and regulatory consequences.

Every production launch needs a rollback plan. If the feature breaks after launch, how quickly can the team revert? Morgan must define the kill switch policy before going live.
